Software Requirements Specification (SRS)

Habit Tracker Web Application
Version 1.0
Date: January 3, 2026

1. Introduction

1.1 Purpose of the System

The Habit Tracker is a web-based application designed to help individuals build, monitor, and maintain positive daily habits. By providing intuitive tools for habit creation, completion tracking, progress visualization, and streak monitoring, the system empowers users to improve personal productivity, health, mindfulness, and overall well-being through consistent behavioral reinforcement.

This document outlines the functional and non-functional requirements, system scope, and technical context to guide development, testing, and future enhancements.

1.2 Project Scope

In Scope

  • Web-based habit creation with customizable categories (e.g., Health, Fitness, Learning).
  • Daily habit completion tracking with persistent storage.
  • Real-time calculation and display of current streaks (consecutive completion days).
  • Weekly calendar view showing completed, missed, and current-day statuses.
  • Overall progress metrics (completion rate, total habits).
  • Responsive user interface accessible on desktop and mobile devices.
  • Client-side data persistence using browser localStorage.
  • Toast notifications for user feedback.

Out of Scope

  • User authentication or multi-user support.
  • Cloud synchronization or backup.
  • Habit reminders or push notifications.
  • Analytics beyond weekly progress (e.g., monthly/yearly trends).
  • Social sharing or community features.
  • Integration with third-party calendars or health apps.

2. Overall System Description

2.1 Intended Users

  • Primary Users: Individuals seeking to develop or maintain daily habits (e.g., students, professionals, fitness enthusiasts, mindfulness practitioners).
  • Technical Profile: Non-technical end users with basic web browser proficiency. No installation or account creation required.

2.2 Operating Environment

  • Client-Side: Modern web browsers (Chrome, Firefox, Safari, Edge – latest two versions). Responsive design supporting mobile (≥320px width), tablet, and desktop viewports.
  • Internet connection required only for initial page load (CDN-hosted assets); app functions offline thereafter.
  • Server-Side: None. The application is fully client-side with no backend server.
  • Data Storage: Browser localStorage (persistent but limited to ~5–10 MB per origin).

2.3 General System Functionality

The Habit Tracker enables users to:

  1. Create new habits by providing a name and selecting a category.
  2. Mark habits as completed each day with a single click.
  3. View all active habits in a scrollable list with individual streak counts.
  4. Monitor weekly progress via a color-coded calendar (green = completed, red = missed, outlined = today).
  5. Track overall statistics, including total habits, current max streak, and daily completion rate.
  6. Delete habits they no longer wish to track.
  7. Receive instant feedback via toast notifications for key actions.

All data is saved automatically in the browser and persists across sessions until cleared by the user or browser.

3. Functional Requirements

ID Requirement Description
FR-01Habit CreationThe system shall allow users to create a new habit by entering a name and selecting a category from a predefined list.
FR-02Habit CompletionThe system shall enable users to mark a habit as completed for the current day. Duplicate completions on the same day shall be prevented.
FR-03Habit DeletionThe system shall allow users to delete a habit permanently after confirmation.
FR-04Streak CalculationThe system shall calculate and display the current consecutive completion streak (in days) for each habit, ending on the current day.
FR-05Daily Completion RateThe system shall compute and display the percentage of habits completed today out of total active habits.
FR-06Weekly Calendar ViewThe system shall display a 7-day calendar (Monday–Sunday) showing:
– Completed days (green)
– Missed past days (red)
– Current day (outlined)
FR-07Data PersistenceAll habit data (name, category, completion dates) shall be saved in browser localStorage and restored on page reload.
FR-08User FeedbackThe system shall display contextual toast notifications for successful actions or errors.
FR-09Responsive LayoutThe UI shall adapt to mobile, tablet, and desktop screen sizes.
FR-10Habit Count DisplayThe system shall show the total number of active habits (e.g., “3 habits”).

4. Non-Functional Requirements

ID Requirement Description
NFR-01PerformanceUI shall render within <500ms on devices with ≥2GB RAM. Actions shall feel instantaneous.
NFR-02UsabilityInterface shall be intuitive for first-time users. WCAG 2.1 AA contrast ratios maintained.
NFR-03ReliabilityData shall not be lost during normal browser navigation or refresh.
NFR-04SecurityAll data remains on the user’s device. No external data transmission beyond CDN assets.
NFR-05CompatibilityShall function on latest Chrome, Firefox, Safari, and Edge.
NFR-06MaintainabilityCode shall be modular, commented, and use CSS variables for theming.
NFR-07AccessibilitySemantic HTML, keyboard navigability, and non-color-dependent status indicators.

5. Assumptions and Dependencies

  • Users understand the concept of “habits” and daily tracking.
  • The device clock is set correctly; streaks depend on accurate system date.
  • localStorage is enabled in the user’s browser.
  • External dependency: Font Awesome 6.4.0 (loaded via CDN).
